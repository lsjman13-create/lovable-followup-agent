# 사용자 관점 문제사항

> 작성: 2026-05-24
> 대상 커밋: `335ab7e` 이후 + Notion Inbox 본문 블록 읽기 확장

---

## 사용자가 적은 4가지 문제

> 개인적으로 사용자관점에서 느끼는 문제라고 생각되는 것들을 정리해봤어.

### (1) LLM 문제

- 물론 현재 API를 사용하지 않은 상황이지만, 생각보다 내용정리가 안됨.
- 더 좋은 모델을 사용하면 What/Context/Assignee/Due Date가 정확히 분리될 수 있음.
- 그러나 Rule 같은 방법을 통해, AI가 판단하는데 더 도움을 줄 수 있음.

#### Claude 분석 — ✅ 일치

코드를 보면 `lovable_agent/process/claude_cli_client.py` 의 `SYSTEM_PROMPT`, 그리고 `lovable_agent/process/ollama_client.py` 의 `SYSTEM_PROMPT` 가 거의 동일한데, 규칙이 사실상 3줄뿐:

1. 잡담·이미지·이모티콘은 무시
2. 비슷한 업무 있으면 `is_duplicate_of` 설정
3. 중복 아니면 `null`

**부족한 부분**

- few-shot 예시 0개 (좋은 추출 / 나쁜 추출 사례 안 보여줌)
- "due_date 가 명시 안 됐으면 추측 금지" 같은 강한 제약 없음
- "assignee 가 '나'면 누구인지 모호하니 그대로 둬라" 같은 카톡 특유 규칙 없음
- 출처 메시지 인용 강제 없음 → 어디 메시지에서 이 task 가 나왔는지 추적 불가
- 두 클라이언트가 같은 프롬프트를 복사로 갖고 있음 (유지보수 시 두 곳 같이 바꿔야 함)

#### 해결 방안

| # | 방안 | 노력 |
|---|---|---|
| a | 프롬프트 공통화 — `lovable_agent/process/prompts.py` 1곳에 정의 | 0.5h |
| b | few-shot 추가 — "이런 카톡 → 이런 task" 예시 3~5개 | 1h |
| c | 규칙 강화 — due_date 추측 금지, assignee 1인칭 처리, title 동사 끝, 반복 통합 | 1h |
| d | 출처 인용 — `ExtractedTask.source_messages` 필드 + 프롬프트에서 메시지 인덱스 강제 | 2h |
| e | 검증 LLM 2회 호출 — 첫 추출 → 두 번째는 "맞는지 검증" (Ollama는 부담 적음) | 2h |
| f | 모델 자체 교체 — `exaone3.5:7.8b` (한국어 강화) 또는 ClaudeCLI/AnthropicAPI | 0.5h |

### (2) 검증 문제

> 현재 Agent가 Tasks에 해야 할 일을 정리해 두더라도, 모든 Task가 들어왔는지는 알 수 없음.

#### Claude 분석 — ✅ 일치

코드 어디에도 "원본 대화에서 누락된 task 가 있는지" 검증하는 기제가 없음. `lovable_agent/process/extractor.py` 가 LLM 응답을 받아서 그대로 노션에 INSERT 하는데, `ExtractionOutcome(new_task_ids, merged_task_ids, raw_extracted_count)` 는 로그에만 남고 사용자가 볼 방법 없음.

매니저가 원본 카톡 다시 읽으면서 비교해야만 누락 확인 가능 — 근데 그러려면 자동화의 의미가 사라짐.

#### 해결 방안

- **a. Inbox row 에 추출 결과 요약 컬럼 추가** — "추출된 업무 수" (number), "추출된 task page 링크" (relation). 클릭 한 번에 결과로 점프
- **b. LLM 에 "이 대화의 핵심 주제 한 줄 요약" 도 같이 출력**시켜 Inbox row 에 저장 → 매니저가 빠르게 판단
- **c. 추출 신뢰도(confidence) 0~1 점수** 같이 받기 → 낮은 점수면 알림
- **d. 정기 회고** — 주 1회 "추출 N건 / 매니저 수정·삭제 비율" 메트릭
- **e. 누락 탐지 자체는 매우 어려움** (LLM 이 못 본 걸 어떻게 알지?) → a+b 로 "사람이 빠르게 확인할 수 있게" 보조하는 것이 현실적

### (3) 데이터 문제

- 일정 이상의 데이터를 넣었을 때 처리하는 시간과 비용이 너무 많이 듬.
- export한 데이터는 채팅창의 모든 대화내역을 기록하므로, 이를 올려서 읽기에는 너무 오랜 시간이 걸릴 것임. 사람 수가 늘어난다면 과부하가 올 가능성이 있음.
- **날짜 단위로 끊어서 분석하도록 만드는 것이 하나의 방법이 될 수 있음.**

#### Claude 분석 — ✅ 일치 (사용자 제안 채택 가능)

사용자가 제안한 "날짜 단위 분할" 은 코드상 거의 공짜로 구현 가능. 이유:

- `lovable_agent/ingest/kakao_parser.py` 의 `parse_kakao_text()` 가 이미 날짜 헤더 (`--------------- 2026년 5월 23일 토요일 ---------------`) 를 인식하고 각 `KakaoMessage` 에 `timestamp(datetime)` 필드를 채워줌
- 즉 "날짜별로 묶기" 는 `messages.groupby(date)` 한 줄로 끝

**현재 한계**

| 한계 | 위치 |
|---|---|
| Inbox 경로가 파서를 통과하지 않음 — raw text 그대로 LLM 에 전달 | `integration_inbox_e2e.py`, `main._process_inbox_once` |
| `format_for_llm(messages, max_messages=50)` 는 "최근 50개" 만 — 오래된 메시지 잘림 | `kakao_parser.py:160` |
| 그래서 무식한 `max_input_chars=3000` 절단으로 도망친 상태 | `config.toml`, `extractor.py` |
| llama3 CPU 변동성 ±50% (같은 입력에 240초 vs timeout) | Ollama CPU 추론 특성 |

#### 해결 방안

- **a. 청크 분할 처리** (사용자 제안 채택) — 노션 Inbox 텍스트도 `parse_kakao_text()` 한 번 통과시키고, 카톡 포맷이면 날짜별 청크 → LLM 청크별 호출 → 결과 병합. 비-카톡 포맷은 현행
- **b. 청크 크기 동적 조정** — 하루 메시지 적으면 묶고, 많으면 오전/오후로 쪼개기. 목표 청크당 ~2K자
- **c. 병렬 처리** — ClaudeCLI/AnthropicAPI 는 동시 호출 가능 (Ollama 는 1:1)
- **d. 톡방 단위 캐싱** — 이미 처리한 메시지 timestamp 이후 것만 새로 분석 (`processed_files` 테이블 컨셉을 메시지 timestamp 까지 확장)
- **e. 사람 수 증가 대응** — 톡방별 활동량 기반 polling interval 동적 조정 (5분~1시간)

### (4) 중복 문제

> 이미 기록된 task에 대해 추가로 채팅창에서 이야기한다면, task가 중복해서 들어갈 수 있음. 이를 검증하거나 확인할 수 있는 방법이 필요함.

#### Claude 분석 — ✅ 일치 (절반은 이미 시도 중)

**현재 구현**

- `claude_cli_client._build_prompt()` 에서 "EXISTING_TASKS (진행 중인 업무)" 블록에 노션 active tasks 를 모두 넣어 전달
- LLM 이 비슷하다고 판단하면 `is_duplicate_of=<task_id>` 채워서 응답
- `extractor.py` 가 그 항목은 INSERT 안 하고 `notion.append_task_note()` 로 기존 task 의 Notes 에 맥락만 추가

**약점**

1. 100% LLM 정확도에 의존 → 헷갈리면 새 task 로 INSERT 되어 중복 발생
2. `list_active_tasks()` 의 `page_size=100` → 활성 task 100건 넘으면 LLM 이 일부만 보고 판단 (운영 1~2개월에 도달)
3. 기존 task 많을수록 프롬프트 길어져 비용·시간 부담
4. "같은 사람·같은 마감인데 제목만 다른" 모호한 케이스 처리 약함

#### 해결 방안

- **a. 1차 임베딩 사전 필터링** — DECISIONS 의 "MVP는 임베딩 없이" 를 운영 단계에서 재검토. 신규 추출 title 임베딩 vs 기존 active task 임베딩 cosine 유사도 > 0.85 인 후보만 LLM 2차 질문. 무료 모델: `ollama pull mxbai-embed-large` 또는 `nomic-embed-text`
- **b. `list_active_tasks()` 페이지네이션** — 100건 제한 해제
- **c. LLM 응답에 `duplicate_confidence` + `duplicate_reason` 추가** — 0.5~0.8 모호 케이스는 매니저 알림
- **d. "중복 의심 검토" 노션 view** — Status=검토 대기 + 비슷한 기존 task 있는 것들 한 자리에서 일괄 정리
- **e. 자동 병합은 신중하게** — 잘못 병합 되돌리기 어렵다. 기본은 "메모만 추가" (현재 구현). 매니저 확정 후 수동 트랜잭션 지원

## 추가로 발견한 문제 (코드 보고 판단)

### (5) 노션에 데이터 넣기가 매번 번거로움

- 카톡 익스포트할 때마다 노션 Inbox 새 row 만들고, 본문 영역 클릭해서 들어가서 텍스트 붙여넣고, Processed 체크 해제 → 매번 4~5번 클릭
- 하루 5개 톡방이면 매일 25번 클릭. 매니저가 귀찮아서 안 쓸 가능성 높음
- 또한 본문 영역에 붙여넣어야 하는데 Memo(title) 칸에 붙여넣으면 잘림 — 본인이 첫 검증에서 실수했던 것 그대로 매니저도 할 수 있음

#### 해결 방안

- **a. 폴더 감시 부활** — ARCHITECTURE §4.1 의 Inbox Watcher 가 사실 폴더 감시 컴포넌트인데 실 구현 없음. `watchdog` 라이브러리는 `.venv` 에 이미 깔려있음. `~/lovable-agent/inbox/` 폴더에 `.txt` 떨어뜨리면 자동 처리하는 것이 PRD FR-1.1 원안. 노션 Inbox 는 자유 메모(FR-1.2) 용도로
- **b. 카톡 익스포트 자동화** (이전 대화에서 사용자가 물어본 것) — 카톡 PC UI 의 "대화 내보내기" 메뉴 자동 클릭해서 위 폴더로 직접 저장
- **c. 단축키·트레이 도구** — 카톡에서 텍스트 복사 → 트레이 메뉴 하나로 "지금 클립보드 내용을 Inbox 에 보내기"
- **d. 노션 UI 에서 본문 입력 안내** — README + Inbox DB 첫 row 에 "여기 본문에 카톡 내용 붙여넣으세요" 가이드

### (6) 잘못 추출된 task 의 수정·삭제 워크플로우 부재

- LLM 이 이상한 task 만들면 매니저가 노션에서 직접 status=취소 또는 row 삭제해야 함
- "이걸 왜 잘못 추출했지?" 학습 피드백 루프 없음
- 매니저가 자주 수정해야 하면 시스템 신뢰도 떨어져서 결국 안 쓰게 됨

#### 해결 방안

- **a. Inbox row "Reject" 빠른 버튼** — Inbox DB 에 "Reject Reason" 컬럼 추가하고 매니저가 한 마디 적으면, 그 reason 을 다음 LLM 호출에 "이런 식으로는 잘못 잡지 마라" negative example 로 주입
- **b. Tasks DB Status=취소 비율 추적** — 일정 % 넘으면 알림
- **c. 매니저 수정 패턴 축적** — 자동 학습은 LoRA 필요해서 MVP 어려움. 대신 주간 "자주 수정되는 패턴" 리포트 → 프롬프트 반영

### (7) 발송된 카톡 메시지의 취소·수정 불가

- 한 번 발송되면 본인이 5분 안에 수동으로 카톡에서 삭제하는 수밖에 없음
- AI 가 잘못된 정보 보냈는데 받는 사람이 먼저 본 후면 정정 메시지 또 보내야 함
- 받는 사람 입장에서 "왜 이 시간에 이런 메시지가 왔지" 불쾌할 수 있음

#### 해결 방안

- **a. 발송 직전 매니저 컨펌 단계** — `config.toml` 에 `require_manual_confirm_before_send` 옵션. true 면 발송 큐 항목을 노션에 "발송 예정" 으로 띄우고 매니저 OK 한 것만 실 발송 (안전성↑, 자동화 효과↓)
- **b. 받는 사람 단위 발송 빈도 상한** — `max_followups_per_recipient_per_day = 3` 같이. 초과하면 다음날로 미룸
- **c. 발송 메시지에 상호작용 유도 문구** — "이의 있으면 답장하세요"
- **d. 발송 이력 노션 view** — "오늘 누구한테 뭐 보냈는지" 일목요연

### (8) 데몬이 잘 돌아가는지 매니저가 알 수 없음

- `main.py` 가 백그라운드에서 도는데 매니저가 보는 화면은 노션·카톡뿐
- 데몬이 죽었거나, Ollama 가 꺼졌거나, Notion API rate limit 걸리면 → 알림이 안 옴
- 매니저는 "오늘 알림 왜 안 왔지" 사후 발견

#### 해결 방안

- **a. Notifier 모듈에 실패 경로 연결** — `winotify` 토스트 즉시 알림. 현재 `winotify` 의존성만 있고 실패 경로에 안 붙어 있음
- **b. 일일 요약 알림** — 매일 아침 "어제: 추출 N건, 발송 N건, 실패 0건" 토스트
- **c. 트레이 아이콘** — `pystray` 라이브러리로 시스템 트레이 상태 표시 (녹색=정상, 빨강=오류)
- **d. 헬스체크 엔드포인트** — `http://localhost:8765/health` 작은 서버 → 브라우저로 한 번에 확인

### (9) PII 노출 위험을 매니저가 인지하기 어려움

- 현재 4가지 LLM 백엔드 (Mock / ClaudeCLI / AnthropicAPI / Ollama) — `config.toml` 의 `llm.backend` 한 줄로 바뀜
- Ollama 만 PII 외부 유출 0. 나머지는 카톡 본문이 외부 서버로 전송됨
- 매니저가 별 생각 없이 backend 를 `ollama` → `claude_cli` 로 바꾸면 회사 톡방 대화가 Anthropic 으로 흘러감

#### 해결 방안

- **a. 데몬 시작 로그에 PII 관점 명시** — "현재 LLM = Ollama (PII 보호)" / "현재 LLM = ClaudeCLI (외부 전송 주의)". 이미 클래스 이름은 찍히지만 PII 관점 설명 없음
- **b. Notion Tasks DB 에 PII Mode 컬럼** — 각 task 가 어느 LLM 으로 추출됐는지 표시
- **c. 본문 자동 마스킹** — 전화번호·주민번호·카드번호 패턴, 외부 전송 백엔드일 때만
- **d. 첫 실행 확인 토스트** — `backend != ollama` 면 1회 매니저 확인

---

### (10) 첫 셋업이 어려움

- 매니저 셀프 셋업 7~8단계: Python·uv 설치, 의존성, Notion 토큰, 부모 페이지, `setup_notion.py`, `config.toml`, Ollama 설치+pull, 시작 프로그램... 기술 모르는 매니저는 못 함
- 본인이 셋업한 것을 그대로 넘기는 시나리오면 OK 지만, 다른 매니저가 새로 도입하려면 진입 장벽 큼

#### 해결 방안

- **a. `setup_wizard.py` 스크립트** — 대화형 셋업: 노션 토큰 → 부모 페이지 → `setup_notion.py` 자동 → `config.toml` 자동 → Ollama 안내 → 끝
- **b. 단일 실행 파일 패키징** — `PyInstaller` 로 `lovable-agent.exe` 만들어서 더블클릭 실행
- **c. README 단계별 스크린샷 가이드**
