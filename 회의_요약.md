## 1. 한 줄로 요약하면

> **카톡에 흩어진 업무를 AI가 노션에 자동으로 정리해주고, 마감일이 다가오면 담당자에게 자동으로 카톡 리마인드를 보내주는, 매니저 1명을 위한 보조 시스템.**

---

## 2. 왜 만드는가 (3가지 문제)

| 매니저가 매일 겪는 문제 | 이 시스템이 하는 일 |
|---|---|
| 📱 정보가 **카톡·이메일·회의·노션**에 흩어져 있음 | 카톡 대화를 AI가 읽고 한곳(노션)에 정리 |
| 🔔 카톡 안에서도 **업무·잡담이 섞여** 며칠 전 지시사항 찾기 어려움 | "할 일 / 맥락 / 마감일 / 담당자" 4가지를 AI가 뽑아냄 |
| ⏰ 캘린더에 기한은 있어도 **누가 무엇을 챙겨야 하는지** 빠짐 | 마감일이 오면 담당자에게 카톡 자동 발송 |

> 💡 **목적**: 매니저가 매일 카톡·노션을 일일이 대조하는 시간을 줄이고, 업무 누락을 막는 것.

## 3. 어떻게 작동하는가 (사람의 시점)

```
[매니저]                                              [담당자]
   │                                                    ▲
   │ ① 카톡 대화 .txt 파일                              │
   │    노션 폴더에 떨굼                                │
   ▼                                                    │
┌──────────────────────────────────────────┐            │
│        AI 에이전트 (매니저 PC에 상주)      │            │
│                                            │            │
│  ② AI가 4요소 추출                        │            │
│     (무엇을 / 왜 / 언제까지 / 누가)        │            │
│                                            │            │
│  ③ 노션 Tasks에 "검토 대기"로 올림         │            │
│                                            │            │
│  ⑤ 마감일이 가까워지면                     │            │
│     ⑥ 자동으로 카톡 발송 ──────────────────┼────────────┘
│        "[AI 자동 팔로우업] 내일까지 X 부탁드립니다"
└──────────────────────────────────────────┘
   ▲
   │ ④ 매니저가 노션에서 확인 → "확정"으로 변경
   │   (확정 안 하면 자동 발송 X)
   │
[매니저]
```

### 핵심 5단계 흐름
1. **모은다** — 매니저가 카톡 대화 텍스트 파일을 노션 폴더에 올림
2. **분석한다** — AI(Claude)가 대화에서 업무 4요소(What / Context / Due Date / Assignee)를 추출
3. **정리한다** — 노션 Tasks DB에 "검토 대기" 상태로 올림
4. **매니저가 확정한다** — 매니저가 확인하고 "확정" 표시 (안 하면 자동 발송 안 됨)
5. **알린다** — 마감 시점에 담당자가 있는 카톡방에 자동 메시지 발송

## 4. 기술 구조 (블록 단위로 본 시스템)

> 큰 그림: 매니저 PC에 **백그라운드 데몬**(=조용히 돌아가는 프로그램) 하나가 있고, 거기서 노션·AI·카톡을 다 조작합니다.

```
┌──────────── 매니저 PC ────────────┐
│                                    │
│  ① Watcher  ─→ 새 카톡 .txt 감지   │
│         │                          │
│         ▼                          │
│  ② Extractor ─→ AI에게 분석 요청 ──┼──→ ☁️ Anthropic Claude API
│         │                          │
│         ▼                          │
│  ③ Repository ─→ 업무 저장 ───────┼──→ 📋 Notion DB
│         │                          │
│         ▼                          │
│  ④ Scheduler ─→ 매분 마감 체크    │
│         │                          │
│         ▼                          │
│  ⑤ Sender ─→ 카톡 PC 자동 조작 ───┼──→ 💬 KakaoTalk PC
│                                    │
└────────────────────────────────────┘
```

| 블록 | 역할 | 풀어쓰면 |
|---|---|---|
| ① Watcher | 폴더 감시 | "새 파일이 들어왔나?" 매분 확인 |
| ② Extractor | AI 분석 | Claude에게 "이 대화에서 업무를 뽑아줘" 요청 |
| ③ Repository | 저장 | 노션(사용자가 보는 곳) + SQLite(내부 상태) 둘 다 관리 |
| ④ Scheduler | 시계 | 매분 "지금 보낼 메시지 있나?" 확인 |
| ⑤ Sender | 카톡 발송 | 카톡 PC 창을 직접 조작해서 메시지 입력·전송 |

## 5. 가장 어려웠던 부분 — 카톡 자동 발송

### 왜 어려운가?
- 카카오는 외부 프로그램이 카톡에 메시지 보낼 수 있는 **공식 통로(API)를 제공 안 함**
- → 어쩔 수 없이 **사람이 카톡 창을 클릭/타이핑하는 것을 흉내내는 방식**(=UI 자동화)을 사용
- 카톡 업데이트로 화면이 바뀌면 깨질 수 있음

### 어떻게 풀었나?
- **오픈소스 참고**: [kakao-sender (v2)](https://github.com/TurnaboutHero/kakao-sender-v2)라는 카톡 자동화 오픈소스 프로젝트의 설계를 차용
- **본인 PC 실측**: 카톡 화면 구조를 실제로 뜯어보고(`investigate.py`) 값을 확정
- **5번 연속 발송 성공** (2026-05-23, "나와의 채팅" 대상)

### 🚨 가장 중요한 안전장치 — "오발송 방지 3중 방어"
**잘못된 사람에게 자동 메시지가 가면 사고**이므로, 3중으로 막습니다.

| 방어선 | 무엇을 검증 | 막아주는 사고 |
|---|---|---|
| ① 활성 탭 강제 | 발송 전 카톡 화면을 명확히 리셋 | 직전에 열려있던 엉뚱한 창에 입력하는 사고 |
| ② 창 비교 | 톡방 열기 전후 창 목록 비교, 새로 열린 1개만 인정 | 다른 채팅창에 잘못 입력하는 사고 |
| ③ 제목 완전일치 | 발송 직전 채팅창 제목이 정확히 일치하는지 재확인 | 동명 톡방·검색 첫 결과에 맹목 발송하는 사고 |

→ **세 단계 모두 통과해야만** 실제 메시지 입력·전송이 일어납니다.

### 그 외 사고 방지 장치
- ✅ **화이트리스트**: 매니저가 미리 "이 톡방에는 자동 발송해도 됨"이라고 등록한 곳에만 발송 (상사·외부 인사 자동 제외)
- ✅ **메시지 접두어**: 모든 자동 메시지 앞에 `[AI 자동 팔로우업]` 강제 표시 (수신자가 "사람이 보낸 게 아님"을 명확히 인지)
- ✅ **검토 대기 단계**: AI가 추출한 업무는 매니저가 "확정"을 눌러야만 자동 발송 활성화
- ✅ **6시간 룰**: PC가 꺼져있던 동안 지나간 알림이 6시간 넘었으면 자동 발송 안 하고 매니저에게만 표시 (받는 사람이 당황하지 않게)
- ✅ **비용 상한**: Anthropic API 월 30,000원 상한 설정 (버그로 비용 폭주 방지)

- 실시간 카톡 자동 수신 (카톡 PC가 메시지를 암호화 저장해서 불가능)

## 9. 핵심 의사결정 (왜 이렇게 했는가)

| 결정 | 왜 |
|---|---|
| **카톡만 먼저** | 노이즈가 가장 크고 누락 위험이 집중된 곳부터 검증 |
| **수동 .txt 업로드** | PC 카톡이 메시지를 암호화 저장 → 자동 수신 불가 |
| **노션 사용** | 팀이 이미 쓰고 있어 학습 비용 0 |
| **검토 대기 단계** | AI 오추출로 엉뚱한 사람에게 메시지 가는 사고를 사람이 한 번 거름 |
| **화이트리스트 방식** | 기본은 차단, 명시적으로 허용한 톡방만 발송 (상사·외부 인사 보호) |
| **`[AI 자동 팔로우업]` 접두어** | 수신자가 "사람이 닦달하는 게 아님"을 알게 해 감정 마찰 완화 |
| **Anthropic Claude** | API가 자동화에 적합, 월 30,000원 상한으로 사고 차단 |
| **Windows PC 상주** | 카톡 PC 자동화가 필요해 클라우드 단독 운영 불가 |

## 10. 위험 요소 (회의에서 질문 받을 수 있는 것)

| 위험 | 대응 |
|---|---|
| 카톡 PC 업데이트로 자동화 깨짐 | 진단 도구(`investigate.py`)로 재실측 후 값만 교체 가능하게 설계 |
| AI가 담당자/기한 잘못 추출 | "검토 대기" 단계에서 매니저가 확정 전엔 발송 불가 |
| 엉뚱한 톡방으로 메시지 누출 | 화이트리스트 + 오발송 3중 방어로 차단 |
| API 비용 폭주 | Anthropic Console에서 월 30,000원 상한 강제 |
| 회사 보안정책 충돌 | AI 호출부를 격리 설계 — 사내 LLM이나 로컬 LLM으로 교체 가능 |
| 동명 톡방 오발송 | "채팅창 제목 완전일치" 검증 (방어선 ③) |

## 부록. 발표 시 강조하면 좋을 한 문장

> "**이번 MVP의 핵심은 자동화의 속도가 아니라, 잘못된 메시지가 가지 않게 막는 안전장치**입니다. 그래서 가장 위험한 카톡 자동 발송 부분을 가장 먼저(Phase 2) 검증하고, 3중 방어 + 화이트리스트 + 검토 대기 + 접두어까지 네 겹으로 사고를 막도록 설계했습니다."
