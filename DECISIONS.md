# Lovable 업무 팔로업 에이전트 — 결정 사항 (입문자용 정리)

> 작성일: 2026-05-22
> 이 문서는 AI 에이전트 시스템을 처음 만들어보는 입장에서 이해할 수 있도록
> 용어를 풀어서 설명한 버전입니다.

---

## 0. 먼저, 단어 정리 (5개만)

| 단어 | 뜻을 풀면 |
|---|---|
| **에이전트(Agent)** | 사람이 시키지 않아도 정해진 규칙대로 알아서 일해주는 작은 프로그램. 여기선 "카톡을 읽고 정리해서 노션에 넣고, 시간 되면 알림을 보내주는 자동 비서" 정도. |
| **LLM (Claude API)** | AI에게 글을 보내면 글로 답해주는 서비스. 여기선 "긴 카톡 내용을 던지면 핵심을 뽑아 정리해주는 똑똑한 비서" 역할. 사용한 만큼 돈을 냄. |
| **API** | 다른 서비스(노션·구글·AI)와 우리 프로그램이 자동으로 정보를 주고받기 위한 통로. 사람이 화면 누르는 대신 프로그램이 누르는 셈. |
| **UI 자동화** | 사람이 마우스·키보드로 하는 일(카톡 켜고, 톡방 클릭하고, 글 쓰고, 엔터)을 프로그램이 대신 하도록 만드는 기술. |
| **DB(데이터베이스)** | 정보를 표 형태로 정리해서 저장하는 곳. 여기선 노션의 표가 곧 DB. |

이 정도만 알면 아래 문서를 읽는 데 무리 없습니다.

## 1. 이 시스템이 결국 뭐 하는 건가요? (한 단락)

지금은 카톡·이메일·회의에서 업무가 쏟아지는데, 그걸 매니저가 일일이 들여다보면서 "누가, 뭘, 언제까지 해야 하지?"를 정리하고, 시간 되면 "이거 마감인데 어떻게 됐어요?" 하고 챙기는 데 너무 많은 시간이 듭니다.

이 시스템은 그 일을 대신해주는 자동 비서입니다. 카톡 대화를 받아서 AI에게 읽히고, 거기서 "할 일 / 맥락 / 마감일 / 담당자"를 뽑아 노션 표에 자동으로 적어주고, 마감이 다가오면 담당자에게 카톡으로 알아서 알림 메시지를 보내줍니다.

---

## 2. 전체 흐름 (그림 한 장으로)

```
  [카톡 대화 .txt 파일]
          │
          │ 사용자가 폴더에 떨군다
          ▼
  ┌─────────────────────┐
  │  ① 읽기 (Ingest)    │  ← "새 파일 들어왔네?" 감지
  └─────────┬───────────┘
            ▼
  ┌─────────────────────┐
  │  ② 이해 (AI 분석)   │  ← Claude AI가 4가지를 뽑아냄
  │  - 할 일             │     · 할 일 (What)
  │  - 맥락              │     · 맥락 (Context)
  │  - 마감일            │     · 마감일 (Due Date)
  │  - 담당자            │     · 담당자 (Assignee)
  └─────────┬───────────┘
            ▼
  ┌─────────────────────┐
  │  ③ 저장 (Notion DB) │  ← 노션 표에 한 줄씩 추가
  └─────────┬───────────┘
            │
            │ 마감일 가까워지면
            ▼
  ┌─────────────────────┐
  │  ④ 알림 (카톡 발송) │  ← PC 카톡을 자동 조작해서
  │                     │     담당자에게 "[AI 자동 팔로우업]
  │                     │     마감 다 됐어요" 메시지
  └─────────────────────┘
```

크게 **읽기 → 이해 → 저장 → 알림** 네 단계입니다.

### 같은 업무가 또 들어오면 어떻게 알아채나?

새로운 카톡 .txt를 분석할 때, 노션의 "진행 중인 업무 목록"(제목·담당자·마감일 정도만)을 **AI 프롬프트에 같이 넣어줍니다**. 그러면 AI가 "이거 새 일이야, 아니면 ID 17번 그 일이야"를 한 번에 판단해서 알려줍니다.

- 새 일 → 노션에 새 줄 추가
- 기존 일 → 그 줄에 메모만 보태기

별도의 "유사도 비교 기술(임베딩)" 같은 건 MVP에선 쓰지 않습니다. 활성 업무가 수십 개 수준이라 AI가 한 번에 비교해도 충분히 빠르고 똑똑합니다. 나중에 활성 업무가 수백~수천 개로 늘면 그때 도입 고민.

## 3. 어디까지 먼저 만들 건가요? (MVP)

MVP는 "최소한 이건 돼야 의미 있다" 하는 첫 번째 완성본을 뜻합니다. 우리 MVP는:

**카카오톡 한 채널만 먼저 한다.**

| 채널 | 1차에서 다룬다? | 이유 |
|---|---|---|
| 카카오톡 | ✅ | 가장 시끄럽고, 가장 많이 놓치는 곳. 여기가 되면 게임 끝. |
| Gmail | ❌ (2차) | 표준 통로(API)가 잘 갖춰져 있어 나중에 붙이기 쉬움. |
| Google Calendar | ❌ (2차) | 위와 동일. |
| Alt(회의록 앱) | ❌ (2차) | 이 앱이 외부에 정보를 내보낼 수 있는지부터 확인 필요. |
| 오프라인 메모 (수동 입력) | ✅ | 사용자가 직접 메모창에 적어 넣는 통로는 1차에 포함. |

가장 어렵고 가치 큰 부분(카톡)부터 검증하고, 그게 잘 돌아가면 다른 채널을 붙입니다.

## 4. 카톡은 어떻게 다루나요?

카톡은 일반 사용자가 외부 프로그램으로 메시지를 직접 읽거나 쓸 수 있는 공식 통로가 없습니다. 그래서 다음과 같이 우회합니다.

### 4.1 읽기 — "텍스트 파일로 내려받아서 폴더에 떨군다"

- 카톡 PC/모바일에서 특정 채팅방을 골라 **"대화 내보내기 → 텍스트 파일(.txt)"** 기능을 사용
- 그 파일을 정해진 폴더에 떨궈두면, 에이전트가 "어, 새 파일이네?" 하고 자동으로 읽음
- (왜 이 방식인가) 카톡 PC가 메시지를 암호화해서 저장하기 때문에 프로그램이 직접 읽기가 매우 어렵고, 화면을 캡처해서 글자를 인식하는 방식은 자주 틀림. 그래서 가장 안정적인 "수동 내보내기" 방식 선택.

### 4.2 쓰기(알림 발송) — "PC 카톡을 프로그램이 직접 조작한다"

- 내 PC에 켜져 있는 카톡 데스크톱을 프로그램이 마치 사람처럼 조작
- 기술: **win32 HWND 주력 + UIA 보조**
  - win32 (`pywin32`): `EnumChildWindows`, `SendMessage(WM_SETTEXT)`, `GetWindowText`, `GetClassName` 으로 카톡 채팅창 HWND와 표준 `RICHEDIT50W` 입력창을 직접 다룸 — 클립보드·포커스 상태에 독립
  - UIA (`uiautomation`): 상태 감지(메시지 리스트 확인) 등 보조용
- 카카오의 "비즈니스/알림톡" 같은 공식 API는 일반 톡방에 못 보내서 못 씀
- **오발송 방지 3중 방어**: ① 친구 탭 강제 ② 발송 직전 카톡 top-level HWND 스냅샷 diff ③ 채팅창 제목 완전일치 — 모두 통과해야만 메시지 입력·전송 진행
- **참고**: 오픈소스 [kakao-sender (v2)](https://github.com/TurnaboutHero/kakao-sender-v2) Phase 3 실측에서 "UIA 단독 한계 → win32 HWND 주력 + UIA 보조"로 갱신된 설계를 그대로 차용. ARCHITECTURE.md §4.6.1~§4.6.5 참조

### 4.3 발송 대상 관리 — "기본은 다 차단, 허락한 톡방에만 보냄"

- 노션에 "AI 자동 메시지 보내도 되는 톡방" 목록(화이트리스트) DB를 따로 둠
- 그 목록에 적힌 톡방에만 자동 발송
- **목록에 없는 톡방의 일은 → 나에게만 알림이 뜨고, 자동 카톡은 절대 안 나감**
- (왜) AI가 자동으로 메시지 보내면 안 되는 어른/상사/외부 인사 관계가 있기 때문. 사고 방지를 위해 "허용한 것만"이 원칙.

## 5. 메시지 문구 규칙

자동으로 나가는 모든 카톡 메시지의 **맨 앞에 반드시** 다음 문구를 붙입니다.

```
[AI 자동 팔로우업]
```

(대괄호와 한 칸 공백 포함)

**왜?** 받는 사람이 "아, 사람이 직접 닦달하는 게 아니라 자동 알림이구나" 하고 부담을 덜 느끼게 하기 위함. 매니저-담당자 관계의 감정적 마찰을 줄이는 장치입니다.

## 6. 결과는 어디에 보이나요? — Notion

정리된 업무는 **노션(Notion)** 에서 봅니다. 자체 웹 대시보드를 따로 만들지 않습니다.

(왜) 팀이 이미 노션을 쓰고 있어서 새로 익힐 게 없음.

### 노션에 만들 표 3개

**(1) Tasks DB — 업무 목록**

| 컬럼 | 무엇을 적나 |
|---|---|
| Title | 업무 요약 한 줄 |
| What | 구체적으로 뭘 해야 하는지 |
| Context | 어떤 맥락에서 나온 일인지 |
| Due Date | 마감일 |
| Assignee | 담당자 |
| Source | 어디서 온 일인지 (`카톡` / `메일` / `캘린더` / `수동 메모`) |
| Source Detail | 톡방 이름, 메일 제목 등 출처 정보 |
| Status | `검토 대기` / `확정` / `진행 중` / `완료` / `취소` |
| AI Followup Enabled | 자동 알림 허용 톡방이면 자동으로 ✔ |
| 발송 이력 | 언제 어떤 메시지를 보냈는지 기록 |

**(2) Chatroom Whitelist DB — 발송 허용 톡방 목록**

| 컬럼 | 무엇을 적나 |
|---|---|
| 톡방명 | 사람이 보는 이름 |
| 카톡 윈도우 타이틀 | 프로그램이 찾을 때 쓰는 정확한 이름 |
| 비고 | 메모 |

**(3) Inbox — 수동 메모/.txt 업로드를 받는 곳**

`Source` 컬럼을 "어디서 왔는지" 표시하도록 설계해두는 이유는, 나중에 이메일·캘린더가 추가돼도 같은 표에 자연스럽게 섞을 수 있게 하기 위함입니다.

## 7. 어디서 돌아가나요? — 내 윈도우 PC

- 카톡 자동 조작이 필요해서 **내 PC가 켜져 있어야** 동작합니다 (클라우드 서버만으론 불가능)
- 부팅하면 자동으로 백그라운드에 떠 있도록 윈도우 작업 스케줄러에 등록
- PC가 꺼져 있을 때 마감이 지나가는 경우는? → 보낼 알림을 잠시 보관(큐잉)했다가 PC 켜지면 처리

(왜 클라우드를 못 쓰나) 카톡은 일반 톡방에 외부에서 자동으로 글 보내는 공식 통로가 없어서, "내 PC의 카톡 창을 직접 조작"하는 게 사실상 유일한 방법.

## 8. 어떤 도구로 만드나요?

| 영역 | 무엇을 쓸 예정 | 풀어서 설명 |
|---|---|---|
| 프로그래밍 언어 | **Python** | AI·노션·카톡 자동화에 필요한 도구가 모두 잘 갖춰진 언어. |
| AI 두뇌 | **Claude API (Anthropic) + 월 30,000원 상한** | 카톡 내용에서 4요소(할 일/맥락/마감일/담당자) 뽑아내는 역할. 일반 작업은 Sonnet, 어려운 건 Opus. 월 상한 설정으로 폭주 사고 방지. (자세한 내용은 8.1) |
| 노션 연동 | `notion-client` | 노션 표에 자동으로 줄 추가·수정. |
| 카톡 자동 조작 | **`pywin32`** (주력) + **`uiautomation`** (보조) | 카톡 PC 메인 창은 UIA 트리가 비어 있어 win32 HWND가 더 안정. 채팅창은 별도 top-level HWND로 표준 `RICHEDIT50W`를 노출 → `SendMessage(WM_SETTEXT)`로 직접 입력. UIA는 상태 감지 보조용. |
| 정해진 시간에 일 시키기 | `APScheduler` | "5분마다 할 일 마감 체크해줘" 같은 타이머 역할. |
| 작은 데이터 저장소 | `SQLite` | 프로그램 안에 쓰는 작은 표. "이 .txt는 이미 처리했음" 같은 메모를 보관. |
| 사용자 메모 입력 | **Notion Inbox 페이지** + 폴더 감시 | 노션의 Inbox 페이지에 자유롭게 메모 작성 → 에이전트가 주기적으로 읽음. .txt는 폴더에 떨구면 자동 처리. |

### 8.1. Claude API를 AI 두뇌로 쓰는 방식 (자세히)

**작동 방식:**

```
[에이전트 Python 코드]
        │
        │ Anthropic SDK로 호출:
        │   client.messages.create(model="claude-sonnet-4-6",
        │                           messages=[...])
        ▼
[Anthropic Claude API]  ← 사용한 토큰만큼 결제
        │
        │ JSON 형식 응답
        ▼
[에이전트가 결과 파싱 → 노션에 저장]
```

**비용 안전장치:**
- **월 상한 30,000원** 설정. 이 금액을 넘으면 API 호출이 자동 정지됨.
- 설정 위치: Anthropic Console → Billing → Usage limits.
- MVP 처리량(하루 .txt 몇 개)이면 보통 월 5,000~15,000원 안팎으로 예상되어, 30,000원은 폭주 사고(예: 버그로 무한 호출) 차단용 한도.

**장점:**
- 프로그램이 자동으로 호출하는 게 본래 용도 → 약관·기술 모두 문제 없음
- 응답이 구조화된 JSON이라 파싱 안정적
- API 직호출이라 속도 빠름

**비용 통제 팁:**
- 카톡 .txt가 길면 토큰이 많이 듦 → 분석 전에 "업무 관련 발화"만 1차 필터링하면 비용 절감
- 대부분 작업은 Sonnet 모델로 처리 (저렴), 정말 어려운 경우만 Opus

### 8.2. 옵션 — "Claude Code 환경 안에서 사용자가 직접 운영하는" 방식 (참고)

> 이 방식은 **현재 MVP에는 채택하지 않으나**, 추후 운영 경험을 토대로 다시 검토할 수 있는 옵션으로 남겨둠.

API 비용·약관 부담을 더 줄이고 싶을 경우, 다음과 같은 대안적 운영 방식이 가능:

- **Claude Code(이 CLI)를 agent의 두뇌로 직접 활용** — 별도 Python 코드가 LLM을 호출하는 게 아니라, 사용자가 Claude Code 세션을 열어 "이 .txt 처리해줘" 같은 자연어 명령으로 운영
- **노션 갱신·중복 판별 등은 Claude Code가 MCP를 통해 직접 처리**
- **시간 기반 자동 발송(Sender)만 별도의 작은 Python 데몬이 담당** — 이쪽엔 AI 호출 없음

**이 방식의 장점**
- Anthropic 약관 위험 0 (Claude Code의 본래 의도된 사용)
- CLI/API 비용 별도 청구 없음 (Claude Code 구독에 포함)
- Sender 데몬 코드가 매우 단순해짐
- 사용자가 분석 결과를 함께 보면서 확정하므로 검토 흐름이 자연스러움

**이 방식의 단점**
- 새 .txt 처리가 완전 자동이 아님 (사용자가 Claude Code를 열어야 분석됨)
- 워치 폴더·트레이 메모창 같은 자동 인입 경로 의미가 없어짐

이 방식으로 전환할 가능성을 염두에 두고, 본 MVP의 **AI 호출부는 한 모듈로 격리해서** 만들어둠. 운영해보고 비용·약관·자동화 요구가 어떻게 균형 잡히는지 본 뒤 재검토.

## 9. "사고 안 나게" 만드는 3가지 안전장치

자동화 시스템에서 가장 무서운 건 "엉뚱한 사람에게 엉뚱한 시간에 엉뚱한 메시지가 나가는 것"입니다. 다음 세 가지로 방지합니다.

| # | 장치 | 막아주는 사고 |
|---|---|---|
| 1 | **AI가 정리한 내용은 일단 `검토 대기` 상태로 둠.** 내가 노션에서 직접 보고 "확정" 누른 것만 자동 알림 대상이 됨. | AI가 담당자나 마감일을 잘못 뽑아서 엉뚱한 사람에게 알림이 가는 사고. |
| 2 | **알림 보내기 직전, 화이트리스트(허용 톡방 목록)를 한 번 더 확인**. | 화이트리스트에서 방금 뺀 톡방에 알림이 새는 사고. |
| 3 | **모든 발송 후 결과를 노션에 기록 + 스크린샷도 같이 저장**. | 안 보냈는데 보낸 줄 알고 넘어가는 사고. 사후에 어떤 메시지가 나갔는지 확인 가능. |

MVP 운영 단계에서 적용할 4가지 정책을 다음과 같이 확정했습니다.

| # | 정한 것 | 한 줄 설명 |
|---|---|---|
| 10-1 | **메모 입력 = 노션 Inbox 페이지** | 오프라인 미팅·이동 중 떠오른 업무는 노션 Inbox 페이지에 자유 형식으로 적음. 에이전트가 주기적으로 읽어 처리. 트레이 앱은 추후 필요시 추가. |

| 10-4 | **늦은 알림 = 6시간 이상 지난 건 자동 발송 스킵** | PC 꺼져 있는 동안 지나간 리마인드는, 예정 시각에서 6시간 이내면 PC 켜질 때 발송, 그 이상 지난 건 자동 발송 안 하고 나에게만 노션·트레이로 표시. |

## 부록: 자주 떠오를 질문에 미리 답하기

**Q. 이 모든 게 돈이 얼마나 들까요?**
- Claude API: 가벼운 사용량 기준 월 5,000~15,000원 예상. 월 상한 30,000원으로 폭주 차단(10-3).
- Notion: 기존에 쓰던 요금 그대로 (추가 비용 0)
- pywin32, uiautomation, Python 라이브러리들: 무료

**Q. 회사 정보(카톡 대화)가 외부로 새지는 않나요?**
- Claude API는 외부 서비스라 카톡 대화 텍스트가 Anthropic 서버를 거칩니다. 회사 정책상 외부 AI 서비스 사용이 제한적이면 (a) 사내 승인된 AI 서비스 사용 (b) 회사 안에서 돌릴 수 있는 로컬 LLM 사용 같은 대안을 별도로 검토해야 합니다.

**Q. 카톡이 업데이트되면 자동화가 깨지지 않나요?**
- 깨질 수 있음. 그래서 화면 좌표가 아니라 win32 HWND(`EnumChildWindows`)와 UIA를 병용해서 요소를 이름·속성으로 찾는다. 그래도 큰 업데이트 후엔 수정이 필요할 수 있어, 진단용 `scripts/investigate.py`로 카톡 HWND 트리를 빠르게 다시 덤프해 확인하는 절차를 마련해둠.

**Q. 처음부터 다 만들지 말고, 더 작게 시작할 수도 있나요?**
- 가능. "AI 분석 + 노션 저장"까지만 먼저 만들고, 카톡 자동 발송은 손으로 하다가 나중에 붙이는 것도 좋은 단계 분할입니다. 정하기 나름.
