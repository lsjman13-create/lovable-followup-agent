# /goal — PLAN 기반 다음 단계 진행

이 프로젝트의 `PLAN.md` 를 단일 진실원(SoT)으로 삼아, 현재 어느 Phase에 있는지·다음에 할 가장 작은 작업이 무엇인지 식별하고, **다음 한 단계를 짧게 발표한 뒤 바로 실행**합니다. 비가역적이거나 외부에 영향이 가는 작업만 사용자에게 한 번 더 확인합니다.

## 사용자 인자

$ARGUMENTS

인자 해석:
- **비어 있음** → 자동으로 다음 단계 식별 후 제안
- `상태` / `status` / `보고` → 현재 진행 상태만 요약 (작업 실행 X)
- `Phase N 시작` → 명시적으로 특정 Phase 진입 제안
- `다시 검토` → 직전 작업의 결과 점검 (커밋·체크박스·README Phase 로그 정합성)
- 그 외 자유 텍스트 → 사용자가 우선하길 원하는 방향으로 해석해 제안

## 처리 순서

### 1. 현재 상태 파악

다음 파일을 순서대로 읽어 컨텍스트를 잡으세요.

1. `PLAN.md` 의 §1 (현재 상태), §3 (Phase별 상세), §6 (체크리스트)
2. `README.md` 의 Phase 로그 표
3. (선택) `git log --oneline -n 10` 으로 최근 커밋 흐름
4. (선택) `git status` 로 미커밋 변경 사항

체크박스 상태와 커밋 이력을 종합해 **현재 진행 중인 Phase**, **이번 Phase에서 완료된 항목**, **다음 후보 작업**을 식별하세요.

### 2. 다음 한 단계 발표

실행 직전, 다음 형식으로 짧게 발표하세요. 항목별로 한두 문장씩만, 스캔 가능하게:

- **현재 Phase**: (예: Phase 1 — 코드 골조 잡기)
- **이번 Phase에서 끝낸 것**: (체크된 항목 목록 또는 "아직 시작 안 함")
- **이번 /goal 에서 할 일**: (구체적인 한 단계 — 파일 한두 개 또는 기능 하나 단위)
- **이 작업을 고른 이유**: (의존 관계, 가장 작은 단위, 위험 큰 부분 우선 등)
- **완료 조건**: (이 작업이 끝났다고 어떻게 판단할지, 측정 가능하게)
- **사전 확인 / 위험**: (작업 진행에 필요한 사전 조건, 알려진 함정)

발표 후 별도 승인을 기다리지 말고 곧장 **§3 실행**으로 진행하세요. 단, 다음 경우에는 발표 후 실행 전에 사용자에게 한 번 더 확인하세요:

- **결정 갈림길**: PLAN과 ARCHITECTURE만으로 답이 안 나오는 트레이드오프가 있을 때 (예: 라이브러리 A vs B 선택)
- **비가역적·외부영향 작업**: §3 실행 가드레일의 항목들
- **PLAN에 없는 새 작업**: 원래 계획에 없던 작업이 끼어들어야 할 때

이 경우엔 "X와 Y 중 어느 쪽이 좋으세요?" 형태로 선택지를 제시하세요.

### 3. 실행

§2에서 발표한 작업을 수행하세요. 실행 중 지킬 것:

- **변경사항 가시화**: 어떤 파일을 만들거나 수정하는지 한 줄로 알리세요.
- **입문자 친화적 설명**: 각 단계가 무엇을 하는 코드·도구인지 1~2문장 설명을 곁들이세요. 라이브러리 선택 이유, 함수 역할 등.
- **외부 서비스 가드레일**: 다음 외부 호출이 필요해지면 PLAN.md의 현재 Phase가 4 미만일 때 **거부**하고 mock으로 대체하는 방안을 제안하세요.
  - Anthropic Claude API 실호출
  - 실제 Notion API (페이지/DB 읽기·쓰기)
  - 실제 카톡 PC 클라이언트에 메시지 전송 (테스트 톡방 PoC는 Phase 2에서 허용)
- **위험한 명령 사전 확인**: 다음 조건이면 실행 전 사용자에게 한 번 더 묻고 진행하세요.
  - 파일 삭제 (`Remove-Item -Recurse` 등)
  - git 비가역 작업 (`reset --hard`, `push --force`, 브랜치 삭제)
  - 환경변수 변경
  - 패키지 매니저 install/uninstall (의존성 추가는 PLAN에 명시된 것 외엔 확인)

### 4. 마무리

작업이 완료되면 다음을 자동으로 처리하세요:

1. **체크박스 갱신**: `PLAN.md` 의 충족된 완료 기준 항목을 `[ ]` → `[x]` 로 바꿉니다.
2. **Phase 로그 갱신**: Phase 가 완료된 경우 `README.md` 의 Phase 로그 표에 완료 날짜와 한 줄 요약을 추가합니다.
3. **새 위험·결정 기록**: 작업 중 발견된 새 위험·결정 사항이 있으면 `PLAN.md` 의 해당 Phase "위험/주의" 또는 `DECISIONS.md` 에 추가합니다.
4. **git 커밋**: 의미 있는 단위로 커밋하세요. 메시지 컨벤션은 다음 절을 따릅니다.
5. **다음 한 줄 요약**: 마지막에 "다음에 할 일: ..." 한 줄로 사용자에게 알립니다.

#### 커밋 메시지 컨벤션

```
<type>: <짧은 요약>

<상세 본문 (선택)>
- 변경한 것
- 왜 변경했는지
- 관련 문서·이슈

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

`type` 후보: `feat` / `fix` / `docs` / `refactor` / `test` / `chore` / `build`

## 원칙 요약

1. **한 번에 한 단계.** /goal 한 번 호출은 작은 단위 하나만 처리.
2. **입문자 친화적.** 라이브러리·패턴이 처음 등장할 땐 한 줄 설명.
3. **외부 의존성은 Phase 4까지 미루기.** Notion·Anthropic·실 발송 등.
4. **체크박스를 건너뛰지 말 것.** 일부만 끝났다면 완료 표시 금지, 블로커 보고.
5. **모르면 묻기.** PLAN/ARCHITECTURE/DECISIONS만으로 결정할 수 없는 트레이드오프는 진행 전 사용자에게 선택지 제시.
6. **위험한 액션은 사전 확인.** 비가역적이거나 외부에 영향이 가는 작업은 한 번 더 묻고 진행.
7. **기본은 바로 실행.** 트레이드오프·비가역·외부영향 외엔 발표 후 즉시 실행. 매번 승인 요구하지 말 것.
