## 1. 한 줄 요약

여러 채널로 흩어진 업무 정보를 노션에 자동 집계하고, 마감이 다가오면 담당자에게 카톡으로 자동 리마인드 메시지를 보내주는 매니저 보조 AI 에이전트.

---

## 2. 배경 (Background)

Lovable 팀 매니저는 MOP, GGE 등 동시다발적으로 운영되는 여러 프로그램의 일정·업무를 관리한다. 현재 다음과 같은 이유로 매일 과도한 매니징 리소스가 소모되고, 일부 업무가 누락될 위험이 존재한다.

1. **소통 채널의 파편화**
   - 정보가 카카오톡(메인), 이메일, 회의, 노션, Google Drive 등에 분산
   - 각 채널을 일일이 확인·대조해야 전체 업무 현황 파악 가능

2. **카톡 채널의 노이즈 및 트래킹 한계**
   - 업무·개인 메시지가 혼재
   - 하루 수십 통의 업무 메시지 중에서 지시사항·회의 내용을 골라내고 기억하는 부담이 극심
   - 'work' 분류만으로는 과거 기록 재확인 시 리소스 큼

3. **일정 관리와 액션 아이템의 괴리**
   - 캘린더에 기한은 있어도 "구체적으로 무엇을 해야 하나"가 명확하지 않음
   - 매일 바쁜 일정 속에서 팔로업 시점을 챙기기 어려움
   - 단순 '일정 기입'을 넘어선 '실행 관리'가 부재

## 3. 목표와 비목표

### 3.1. 목표 (Goals)

- **G1.** 카톡 대화에서 업무를 자동으로 식별·구조화하여 노션에 집계
- **G2.** 마감일이 다가오는 업무에 대해 담당자에게 자동 리마인드 메시지를 발송
- **G3.** AI가 잘못 판단하거나 부적절한 대상에게 메시지가 가는 사고를 구조적으로 차단
- **G4.** 매니저가 매일 카톡·노션을 일일이 대조하는 리소스를 줄이고, 누락 위험을 낮춤

### 3.2. 비목표 (Non-Goals)

MVP에서는 다음을 다루지 않는다.

- **NG1.** Gmail·Google Calendar·Alt 회의록 등 카톡 외 채널 연동 (Phase 2)
- **NG2.** 자체 웹 대시보드 (Phase 2 이후)
- **NG3.** 팀원이 직접 시스템을 사용하는 셀프서비스 (MVP는 매니저 1인 운영)
- **NG4.** 실시간 카톡 메시지 자동 수신 (수동 `.txt` 익스포트로 갈음)
- **NG5.** 사외(고객사·외부 인사) 대상 자동 메시지 발송

## 4. 대상 사용자 (Users)

### 4.1. 1차 사용자: 매니저 본인

- 역할: Lovable 팀 내 매니저
- 사용 빈도: 매일 (시스템과 직접 상호작용)
- 주요 작업: 카톡 `.txt` 업로드, 노션에서 업무 검토·확정, 리마인드 결과 확인
- 기대: "오늘 챙길 것이 한눈에 보이고, 자동으로 담당자에게 알림이 가서 매니징 부담이 준다"

### 4.2. 2차 사용자: 담당자들 (메시지 수신자)

- 역할: 매니저가 관리하는 업무를 실제 수행하는 팀원·동료
- 시스템과 직접 상호작용 없음
- 시스템으로부터 받는 것: 카톡으로 `[AI 자동 팔로우업]` 접두어가 붙은 리마인드 메시지
- 기대: "사람이 직접 닦달하는 게 아니라 자동 알림이라는 게 명확해서 부담이 적다"

### 4.3. 메시지 발송 대상이 아닌 인사

- 역할: 매니저의 상사·외부 인사 등 AI 자동 메시지가 부적절한 관계
- 시스템 상의 처리: 화이트리스트에 등록되지 않아 자동 발송 차단
- 해당 업무는 매니저 본인의 액션 아이템으로만 남고 자동 메시지는 발송되지 않음

## 5. 사용자 시나리오 (User Stories)

### S1. 회의 후 액션 아이템 정리

> 매니저는 30분짜리 카톡 회의를 마치고 5개 톡방의 대화를 .txt로 익스포트한다. 노션 Inbox에 업로드하면 잠시 후 Tasks DB에 새 업무 4건이 `검토 대기` 상태로 떠 있다. 매니저는 4건을 확인해서 3건은 `확정`, 1건은 잘못 추출되어 직접 수정 후 `확정`한다.

### S2. 자동 리마인드 발송

> 어떤 업무의 마감이 내일 오후 3시다. 시스템은 마감 24시간 전(오늘 오후 3시)에 담당자가 속한 톡방이 화이트리스트에 있는지 확인한 후, 자동으로 `[AI 자동 팔로우업] 내일 3시까지 X 업무 진행 부탁드립니다` 메시지를 발송한다. 발송 직후 화면 스크린샷이 노션 발송 이력에 첨부된다.

### S3. AI가 자동 발송 못 하는 대상

> 상사가 매니저에게 부탁한 업무가 있다. 시스템은 그 톡방이 화이트리스트에 없음을 확인하고 자동 발송하지 않는다. 대신 매니저의 노션 Tasks 화면에 "수동 처리 필요" 표시가 뜬다.

### S4. 오프라인 미팅 메모

> 매니저가 외부 미팅 중에 새 업무를 들었다. 매니저는 노션 Inbox 페이지에 "다음 주 수요일까지 김매니저에게 PRD 초안 공유" 식으로 자유롭게 적는다. 다음 처리 사이클에서 시스템이 이 메모를 읽어 Tasks DB의 검토 대기 항목으로 만든다.

### S5. PC가 꺼져 있던 동안의 누락

> 주말 동안 PC가 꺼져 있었다. 월요일 아침 PC를 켜자 토요일 오전 9시 예정 알림은 6시간을 한참 넘었으므로 자동 발송되지 않고 노션에 "수동 처리 필요"로만 표시된다. 일요일 저녁 9시 예정 알림은 12시간 미만이라 일부 발송 가능... → 본 시스템은 보수적으로 처리하여, 6시간 룰 적용. 6시간 초과한 모든 건은 사용자에게만 표시.

## 6. 기능 요구사항 (Functional Requirements)

### 6.1. 데이터 수집 (Input)

- **FR-1.1.** 사용자는 카카오톡에서 익스포트한 `.txt` 파일을 노션 Inbox 페이지 또는 지정 폴더에 올릴 수 있어야 한다.
- **FR-1.2.** 사용자는 노션 Inbox 페이지에 자유 형식 메모를 작성할 수 있어야 한다.
- **FR-1.3.** 시스템은 새로 들어온 `.txt`·메모를 자동으로 감지하여 처리 대기열에 올려야 한다.
- **FR-1.4.** 이미 처리된 같은 `.txt`는 중복 처리되지 않아야 한다.

### 6.2. 분석 및 구조화 (Process)

- **FR-2.1.** 시스템은 인입된 비정형 데이터에서 다음 4요소를 추출해야 한다.
  - What: 구체적 업무 내용
  - Context: 발생 맥락 및 진행 방식
  - Due Date: 최종 처리 기한
  - Assignee: 업무 담당자
- **FR-2.2.** 새 업무 추출 시, 노션의 진행 중인 기존 업무 목록을 같이 참고하여 중복 여부를 판단해야 한다.
- **FR-2.3.** 새 업무는 `Status = 검토 대기` 상태로 노션 Tasks DB에 추가되어야 한다.
- **FR-2.4.** 기존 업무와 중복이라고 판단되면 새 줄을 추가하지 않고 해당 업무에 맥락을 보태야 한다.
- **FR-2.5.** 매니저가 노션에서 `Status = 확정`으로 변경한 업무에 대해서만 자동 리마인드가 활성화되어야 한다.

### 6.3. 출력 — 카톡 자동 리마인드 (Output)

- **FR-3.1.** 시스템은 확정된 업무의 마감일을 보고 적절한 시점(예: D-1 / D-day)에 담당자에게 리마인드 메시지를 발송해야 한다.
- **FR-3.2.** 발송 메시지 텍스트 맨 앞에는 반드시 `[AI 자동 팔로우업] ` 접두어가 들어가야 한다.
- **FR-3.3.** 발송 대상 톡방이 노션의 Chatroom Whitelist DB에 등록되어 있을 때만 자동 발송한다.
- **FR-3.4.** 화이트리스트에 없는 톡방 대상 업무는 노션 Tasks의 해당 행에 "수동 처리 필요" 표시만 남기고 자동 발송하지 않는다.
- **FR-3.5.** 모든 발송 후, 발송 시각·메시지 내용·수신 톡방·스크린샷을 노션 발송 이력에 기록해야 한다.

### 6.4. 지연된 알림 처리

- **FR-4.1.** PC가 꺼져 있던 동안 지나간 리마인드는, **예정 시각에서 6시간 이내**인 경우 PC가 켜진 즉시 발송한다.
- **FR-4.2.** 6시간을 초과한 경우 자동 발송을 스킵하고, 노션과 데스크톱 알림으로 사용자에게만 표시한다.

### 6.5. 사용자 통제

- **FR-5.1.** 사용자는 노션 Chatroom Whitelist DB를 편집하여 자동 발송 허용 톡방을 추가·제거할 수 있어야 한다.
- **FR-5.2.** 사용자는 노션 Tasks DB의 임의 항목에 대해 `AI Followup Enabled`를 끄거나 켤 수 있어야 한다.
- **FR-5.3.** 사용자는 임의의 발송 이력을 노션에서 조회할 수 있어야 한다.

## 7. 비기능 요구사항 (Non-Functional Requirements)

### 7.1. 신뢰성

- **NFR-1.1.** 화이트리스트에 없는 톡방으로 자동 메시지가 발송되는 사고는 발생하지 않아야 한다 (Sev-1).
- **NFR-1.2.** "검토 대기" 상태인 업무에 대해 자동 발송이 일어나지 않아야 한다 (Sev-1).
- **NFR-1.3.** 발송된 모든 메시지에는 `[AI 자동 팔로우업] ` 접두어가 누락 없이 포함되어야 한다 (Sev-1).
- **NFR-1.4.** 발송 성공/실패 여부가 노션 발송 이력에 정확히 기록되어야 한다.

### 7.2. 비용

- **NFR-2.1.** Anthropic Claude API 월 사용량은 **30,000원**을 초과하지 않아야 한다 (Anthropic Console 상한 설정으로 강제).

### 7.3. 보안 / 프라이버시

- **NFR-3.1.** 카톡 대화 텍스트는 Anthropic API에만 전송되며 다른 외부 서비스로는 흘러가지 않는다.
- **NFR-3.2.** 노션·Anthropic 인증 정보는 평문 파일에 저장하지 않고, 환경변수 또는 OS 자격증명 저장소를 사용한다.
- **NFR-3.3.** 발송 이력 스크린샷은 노션 파일로만 저장하며 로컬 디스크에는 임시로만 보관 후 삭제한다.

### 7.4. 운영성

- **NFR-4.1.** 사용자가 Windows PC를 재부팅해도 시스템이 자동으로 다시 동작해야 한다.
- **NFR-4.2.** 시스템 오류 발생 시 데스크톱 알림 + 노션 로그로 사용자에게 즉시 가시화되어야 한다.

## 8. 성공 지표 (Success Metrics)

MVP 운영 1개월 후 다음을 확인한다.

| 지표 | 목표값 |
|---|---|
| 노션 Tasks DB에 자동 추가된 업무 정확도 (4요소 모두 맞음) | ≥ 80% |
| 자동 발송된 리마인드 메시지 중 정확하게 발송된 비율 | 100% (잘못된 대상·내용 발송 0건) |

| 자동 발송 시도 실패율 (카톡 자동화 실패) | < 5% |
| 월간 Anthropic API 비용 | ≤ 30,000원 |

## 9. 범위 (Scope)

- 카카오톡 단일 채널 (수동 `.txt` 익스포트 방식)

### 9.2. Phase 2 이후 검토

- Gmail / Google Calendar 자동 인입
- Alt 등 회의록 앱 연동
- 자체 웹 대시보드
- 팀원 셀프서비스 (다중 사용자)
- 실시간 카톡 자동 수신 (안드로이드 봇폰 등)

## 10. 제약과 가정 (Constraints & Assumptions)

### 10.1. 제약

- **C1.** 카톡은 일반 톡방에 외부 프로그램이 자동 발송할 수 있는 공식 API가 없다 → PC 카톡 UI 자동화 사용
- **C2.** PC 카톡은 메시지를 암호화 저장한다 → 자동 수신 불가, 수동 `.txt` 익스포트 사용
- **C3.** 카톡 자동 발송을 위해 매니저 PC가 켜져 있어야 한다
- **C4.** Anthropic API 사용은 회사 보안정책상 허용된다고 가정한다 (별도 확인 필요)

### 10.2. 가정

- **A1.** 매니저는 매일 1회 이상 카톡 `.txt` 익스포트를 수행할 수 있다
- **A2.** 매니저는 노션을 일상적으로 사용하고 있다
- **A3.** 활성 업무 수는 평균 수십 개 이하 (수백 개를 넘으면 데이터 모델 재설계 필요)

## 11. 주요 위험과 대응 (Risks & Mitigations)

| ID | 위험 | 영향 | 대응 |
|---|---|---|---|
| R1 | 카톡 PC 클라이언트 업데이트로 UI 자동화 깨짐 | 자동 발송 중단 | win32 HWND(`EnumChildWindows`) 주력 + UIA(`uiautomation`) 보조. `WindowSpec`로 식별 정보 캡슐화. Step 단위 추상화로 깨진 단계 즉시 식별. 진단용 `scripts/investigate.py`로 카톡 HWND 트리 재실측 절차 마련. 발송 후 스크린샷 + 상태 감지로 결과 확인. 실패 시 즉시 데스크톱 알림. |
| R2 | AI가 담당자·기한을 잘못 추출 | 엉뚱한 사람에게 발송 | `검토 대기` 상태로 들어오며 매니저 확정 전엔 발송 불가 |
| R3 | 화이트리스트에 없는 톡방으로 메시지 누출 | 부적절한 대상에게 자동 메시지 | 발송 직전 화이트리스트 더블체크. 캐시와 노션 원본 둘 다 확인. |
| R4 | API 비용 폭주 (버그로 인한 무한 호출) | 예상 못한 청구 | Anthropic Console에서 월 30,000원 상한 설정 |
| R5 | Anthropic 약관·요금 정책 변경 | 운영 중단 가능 | AI 호출부를 단일 모듈로 격리. 옵션(로컬 LLM, Claude Code 직접 운영 등)으로 전환 가능 |
| R6 | 카톡 대화 내용의 외부 유출 우려 (회사 보안) | 운영 자체가 막힘 | 사내 정책 사전 확인. 막히면 사내 승인 LLM 또는 로컬 LLM로 변경 |
| R7 | 동명 톡방·검색결과 오인으로 엉뚱한 채팅창에 발송 | 부적절 발송 사고 (Sev-1) | 오발송 방지 3중 방어 적용 — ① 친구 탭 강제 ② HWND 스냅샷 diff(새로 열린 1개만 인정) ③ 채팅창 제목 완전일치. 모두 통과해야만 메시지 입력·전송 (ARCHITECTURE.md §4.6.3) |

## 13. 부록

### 13.1. 참고 결정 사항

- 임베딩 미사용 (Claude 프롬프트에 기존 업무 목록 포함하여 중복 판별) — [DECISIONS.md](DECISIONS.md) §2
- AI 엔진은 Anthropic Claude API 사용, 월 30,000원 상한 — [DECISIONS.md](DECISIONS.md) §10-3
- "Claude Code 환경에서 직접 운영" 방식은 옵션으로 보존 — [DECISIONS.md](DECISIONS.md) §8.2

### 13.2. 용어

- **카톡 .txt 익스포트**: 카카오톡의 "대화 내보내기" 기능으로 생성되는 텍스트 파일
- **화이트리스트**: AI가 자동으로 메시지를 보내도 좋다고 매니저가 허용한 톡방 목록
- **검토 대기**: AI가 추출했지만 매니저가 아직 확인하지 않은 업무 상태
- **확정**: 매니저가 검토 후 자동 리마인드를 활성화한 업무 상태
