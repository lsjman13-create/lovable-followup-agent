개인적으로 사용자관점에서 느끼는 문제라고 생각되는 것들을 정리해봤어.

(1) LLM 문제
- 물론 현재 API를 사용하지 않은 상황이지만, 생각보다 내용정리가 안됨.
- 더 좋은 모델을 사용하면 What/Context/Asignee/Due Date가 정확히 분리될 수 있음
- 그러나 Rule 같은 방법을 통해, AI가 판단하는데 더 도움을 줄 수 있음.

→ Claude 분석:
일치함. 코드를 보면 lovable_agent/process/claude_cli_client.py 의 SYSTEM_PROMPT,
그리고 lovable_agent/process/ollama_client.py 의 SYSTEM_PROMPT가 거의 동일한데,
규칙이 사실상 3줄뿐이야:
  1) 잡담·이미지·이모티콘은 무시
  2) 비슷한 업무 있으면 is_duplicate_of 설정
  3) 중복 아니면 null
부족한 부분:
  - few-shot 예시 0개 (좋은 추출 / 나쁜 추출 사례 안 보여줌)
  - "due_date 가 명시 안 됐으면 추측 금지" 같은 강한 제약 없음
  - "assignee 가 '나'면 누구인지 모호하니 그대로 둬라" 같은 카톡 특유 규칙 없음
  - 출처 메시지 인용 강제 없음 — 어디 메시지에서 이 task 가 나왔는지 추적 불가
  - 두 클라이언트가 같은 프롬프트를 복사로 갖고 있음 (유지보수 시 두 곳 같이 바꿔야 함)

해결 방안:
  a. 프롬프트 공통화 — lovable_agent/process/prompts.py 같은 모듈 1곳에 정의
  b. few-shot 추가 — "이런 카톡 → 이런 task" 예시 3~5개
  c. 규칙 강화:
     - due_date 는 명시된 날짜만, 추측 금지 (애매하면 null)
     - assignee 가 "나/저" 같은 1인칭이면 source_detail(톡방 이름) 그대로 표기
     - title 은 동사로 끝나기 (예: "보고서 작성", "일정 확정")
     - 1개 메시지에서 동일 행위 반복 언급 시 1개로 통합
  d. 출처 인용 — ExtractedTask 에 source_messages 필드 추가하고 프롬프트에서
     "이 task 의 근거가 된 메시지 인덱스를 함께 반환하라" 강제
  e. 결과 검증 LLM 2회 호출 — 첫 호출은 추출, 두 번째는 "이 추출이 맞는지 검증"
     (비용·시간 trade-off — Ollama 면 부담 적음, Claude CLI 면 2배)
  f. 모델 자체 교체 — exaone3.5:7.8b (한국어 강화) 또는 ClaudeCLI/AnthropicAPI


(2) 검증 문제
- 현재 Agent가 Tasks에 해야 할 일을 정리해 두더라도, 모든 Task가 들어왔는지는 알 수 없음.

     a + b 로 "사람이 빠르게 확인할 수 있게" 보조하는 것


(3) 데이터 문제
- 일정 이상의 데이터를 넣었을 때 처리하는 시간과 비용이 너무 많이 듬.
- export한 데이터는 채팅창의 모든 대화내역을 기록하므로, 이를 올려서 읽기에는 너무 오랜 시간이 걸릴 것임. 사람 수가 늘어난다면 과부하가 올 가능성이 있음.
- 날짜 단위로 끊어서 분석하도록 만드는 것이 하나의 방법이 될 수 있음.

→ Claude 분석:
일치함. 그리고 사용자가 제안한 "날짜 단위 분할" 은 코드상 거의 공짜로 구현 가능.
이유:
  - lovable_agent/ingest/kakao_parser.py 의 parse_kakao_text() 가 이미 날짜 헤더
    (--------------- 2026년 5월 23일 토요일 ---------------)를 인식하고
    각 KakaoMessage 에 timestamp(datetime) 필드를 채워줌
  - 즉 "날짜별로 묶기" 는 messages.groupby(date) 한 줄로 끝
현재 한계:
  - Inbox 경로(integration_inbox_e2e.py)가 이 파서를 통과하지 않음.
    노션 본문 블록의 raw text 를 그대로 LLM 에 전달. 그래서 max_input_chars=3000
    같은 무식한 절단을 하게 된 것.
  - format_for_llm(messages, max_messages=50) 함수가 이미 있는데
    "최근 50개" 기준이라 오래된 메시지가 잘림 (Inbox 시나리오와 안 맞음)
  - llama3 CPU 추론 변동성 ±50% — 같은 입력에도 어떤 날은 240초, 어떤 날은 timeout

해결 방안:
  a. 청크 분할 처리 (사용자 제안 채택):
     - 노션 Inbox 텍스트도 parse_kakao_text() 로 한 번 파싱 시도
     - 카톡 포맷이면 → 날짜별로 청크 분할 → 청크마다 LLM 호출 → 결과 병합
     - 비-카톡 포맷(자유 메모)이면 → 현행대로
     구현 위치: TaskExtractor 에 chunk_strategy 파라미터 추가
  b. 청크 크기 동적 조정:
     - 하루 메시지 수가 적으면 여러 날 묶기
     - 많으면 하루를 오전/오후로 더 쪼개기
     목표: 청크당 ~2K자 (llama3 CPU 에서 안정적으로 5분 내 처리)
  c. 병렬 처리 — Ollama 는 1요청 1처리지만, ClaudeCLI/AnthropicAPI 는 동시 호출 가능
  d. 채팅방 단위 캐싱 — 이미 처리한 메시지 timestamp 이후 것만 새로 분석
     (SQLite 의 processed_files 테이블 컨셉을 메시지 timestamp 까지 확장)
  e. 사람 수 증가 대응 — 톡방별 활동량 기반 polling interval 동적 조정 (5분~1시간)


(4) 중복 문제
- 이미 기록된 task에 대해 추가로 채팅창에서 이야기한다면, task가 중복해서 들어갈 수 있음. 이를 검증하거나 확인할 수 있는 방법이 필요함.

(7) 발송된 카톡 메시지의 취소·수정 불가
- 한 번 카톡으로 발송되면 본인이 5분 안에 수동으로 카톡에서 삭제하는 수밖에 없음
- AI 가 잘못된 정보 보냈는데 받는 사람이 먼저 본 후면 정정 메시지 또 보내야 함
- 받는 사람 입장에서는 "왜 이 시간에 이런 메시지가 왔지" 불쾌할 수 있음

해결 방안:
  a. 발송 직전 매니저 컨펌 단계 — config.toml 에 require_manual_confirm_before_send 옵션.
     true 면 발송 큐의 항목을 노션에 "발송 예정" 으로 띄우고 매니저가 OK 한 것만 실 발송.
     안전성 ↑, 자동화 효과 ↓
  b. 톡방·받는 사람 단위 발송 빈도 상한 — config.toml 에
     max_followups_per_recipient_per_day = 3 같이. 초과하면 큐에 넣고 다음날 발송
  c. 발송 메시지에 "이의 있으면 답장하세요" 같은 상호작용 유도 문구 추가
  d. 발송 이력 노션 view — 매니저가 "오늘 누구한테 뭐 보냈는지" 일목요연하게 확인
